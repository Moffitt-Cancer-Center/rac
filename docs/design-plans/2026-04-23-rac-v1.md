# RAC v1 Design

## Summary

RAC (Research Application Commons) is a self-hosted platform that gives research institutions a governed path for deploying researcher-authored containerized applications — replacing the ad-hoc server access and shadow-IT arrangements that researchers currently rely on. Each institution runs a fully independent copy inside its own Azure subscription; there is no shared infrastructure, no multi-tenant database, and no cross-institution logic. Moffitt Cancer Center is the first deployment.

The system is organized around three tiers of infrastructure and three runtime services. Static platform infrastructure — networking, container registry, database, key management, observability — is provisioned once via Bicep IaC and lives at Tier 2. Dynamic per-submission resources (each researcher app, its DNS record, its signing keys) are provisioned at approval time by the Control Plane calling the Azure SDK directly, keeping runtime provisioning out of IaC pipelines (Tier 3). The three runtime services are the Control Plane (FastAPI + React: submission intake, two-stage approval, cost dashboards, token issuance), the Token-Check Shim (a reverse proxy that is the sole public entry point to researcher apps, enforcing JWT reviewer-token validation and handling cold-start wake-up), and the Build + Scan Pipeline (a GitHub Actions workflow that builds images, runs Grype and Defender for Containers, and posts a signed verdict back to the Control Plane). Researcher apps themselves run with internal ingress only and scale to zero when idle; the Shim handles the cold-start interstitial so reviewers see a deliberate wait rather than a browser error.

## Definition of Done

v1 of RAC is *done* when all of the following are true in a single Azure subscription deployment:

1. **Platform is deployable from source.** Running the `infra/` Bicep modules via GitHub Actions against an Azure subscription + Entra tenant produces a working RAC environment. A promotion workflow exists for dev → staging → prod.

2. **Researchers can submit applications end-to-end.** An authenticated researcher (via Entra SSO) can submit a GitHub repo URL + Dockerfile path via the Control Plane web UI; the submission is recorded, the pipeline builds and scans the image, and the submission progresses through the two-stage approval workflow (research leadership → IT).

3. **Submission is API-first.** All submission actions the web UI can perform are available via authenticated REST endpoints. Client-credentials OAuth2 auth is supported alongside interactive OIDC. Idempotency keys are honored on writes. Webhook subscriptions deliver HMAC-signed callbacks to registered agents on status changes.

4. **Pre-submission detection rules operate.** A starter set of rules (inline-download detection, huge-files-in-git, missing `USER`, undeclared assets) run on submission; findings are surfaced as nudges; researcher decisions are recorded with `rule_id@version`.

5. **Build + scan pipeline enforces the severity gate.** The pipeline (in a separate `rac-pipeline` repo) clones the researcher's repo, builds the image with a buildx cache, pushes to ACR, runs Grype + waits for Defender, consolidates findings, and calls back to the Control Plane. The severity gate is configured per-deployment.

6. **Approved apps deploy automatically.** On final approval, the Control Plane provisions a new ACA app (via Azure SDK), creates the DNS record, mints per-app JWT signing keys in Key Vault, mounts declared assets, and brings the app live at `<slug>.${PARENT_DOMAIN}` with `min-replicas=0`.

7. **Reviewer access works via tokenized URLs.** Researchers can mint reviewer tokens (label + expiry) through the Control Plane. The Token-Check Shim validates tokens, sets cookies, serves cold-start interstitials, serves branded error pages for expired/revoked/invalid tokens, and writes per-request audit rows.

8. **Asset handling supports v1 kinds.** The manifest format is parsed; `upload` and `external_url` asset kinds work end-to-end (sha256 verification, mount at runtime). `shared_reference` is structurally supported in the schema but not exposed in the submission UI (deferred to v2).

9. **Ownership model is wired.** Every app has `pi_principal_id` and `dept_fallback`. A nightly job against Microsoft Graph detects deactivated PIs and flags affected apps for departmental review.

10. **Observability is operational.** App Insights + Log Analytics are wired; custom metrics are emitted; pager-tier alerts are configured; structured JSON logs with stable fields go to Log Analytics; Event Hub is documented as the SIEM export point.

11. **Cost attribution works.** Every provisioned resource is tagged with `rac_app_slug`, `rac_pi_principal_id`, `rac_submission_id`; `cost_snapshot_monthly` is populated from Azure Cost Management exports; the admin dashboard shows per-app costs.

12. **Approved automated tests pass.** Control Plane unit + API + integration tests pass; shim unit + integration tests pass; golden-repo tests validate the pipeline against at least one known-clean and one known-bad repo.

**Explicitly out of scope for v1:**

- ServiceNow connector (architecture supports it; adapter not built)
- `shared_reference` catalog UI and curation workflow (data model exists; UI deferred to v2)
- SIEM integration itself (export surface exists; connector deferred)
- Multi-region (never)
- CLI client (`rac submit .`)
- Dry-run submission endpoint

## Acceptance Criteria

### rac-v1.AC1: Platform is deployable from source

- **rac-v1.AC1.1 Success:** `infra-deploy.yml` against a clean dev Azure subscription provisions all Tier 2 resources without manual intervention.
- **rac-v1.AC1.2 Success:** `az deployment sub what-if` on `main.bicep` against an unchanged environment produces zero changes.
- **rac-v1.AC1.3 Success:** Promotion workflow requires human approval before deploying to staging or prod.
- **rac-v1.AC1.4 Failure:** Missing a required deployment parameter (e.g., `PARENT_DOMAIN`) causes deploy to fail with a clear, actionable error message naming the missing parameter.
- **rac-v1.AC1.5 Edge:** All provisioned resources carry the required tags (`rac_env` at minimum) at the moment of creation — not as a follow-up apply.

### rac-v1.AC2: Researchers can submit applications end-to-end

- **rac-v1.AC2.1 Success:** An authenticated researcher submits a valid GitHub URL + Dockerfile path via the web UI; a `submission` row is created with `status=awaiting_scan`, the correct `submitter_principal_id`, and a slug derived from the paper metadata.
- **rac-v1.AC2.2 Success:** A submission progresses through `awaiting_scan → awaiting_research_review → awaiting_it_review → approved → deployed` when the scan passes and both approvers approve, with correct state transitions and timestamps recorded.
- **rac-v1.AC2.3 Failure:** Unauthenticated request to `POST /submissions` returns 401.
- **rac-v1.AC2.4 Failure:** Submission with a GitHub URL that returns 404 surfaces a validation error before the pipeline is dispatched.
- **rac-v1.AC2.5 Failure:** Submission with a missing Dockerfile at the declared path transitions to `pipeline_error` or `scan_rejected` (per pipeline failure mode) with a researcher-visible error message.
- **rac-v1.AC2.6 Edge:** The researcher's Entra object ID (`oid`) is persisted as `submitter_principal_id` and used consistently across `submission`, `approval_event`, and `detection_finding` rows.

### rac-v1.AC3: Submission is API-first

- **rac-v1.AC3.1 Success:** OAuth2 client credentials auth with a registered `agent` grants access to submission endpoints; `agent_id` is recorded on every submission created via this path.
- **rac-v1.AC3.2 Success:** A duplicate `POST /submissions` with the same `Idempotency-Key` returns the original `submission_id` and HTTP 200 (not a new row).
- **rac-v1.AC3.3 Success:** A registered webhook subscriber receives an HMAC-signed callback for every submission state transition matching its subscribed event types.
- **rac-v1.AC3.4 Failure:** Pipeline callback with an invalid HMAC signature is rejected with 401 and the submission state is unchanged.
- **rac-v1.AC3.5 Failure:** Request from a disabled `agent` (via client credentials) returns 403.
- **rac-v1.AC3.6 Edge:** A webhook subscription auto-disables after a configurable number of consecutive delivery failures; operator sees this in the admin UI.

### rac-v1.AC4: Pre-submission detection rules operate

- **rac-v1.AC4.1 Success:** A submission with `RUN wget https://...` in its Dockerfile fires the `dockerfile/inline_downloads` rule and surfaces the finding in the researcher UI with an accept/override choice.
- **rac-v1.AC4.2 Success:** A submission with a file larger than the configured threshold committed to git fires the `repo/huge_files_in_git` rule.
- **rac-v1.AC4.3 Success:** A researcher's decision (accept, override, auto-fix, dismiss) on a finding is persisted with `rule_id`, `rule_version`, decision, and timestamp.
- **rac-v1.AC4.4 Success:** Adding a new rule file to `apps/control-plane/src/detection/rules/` is picked up on service start without modifying registration or orchestration code elsewhere.
- **rac-v1.AC4.5 Success:** A service-to-service submission (via client credentials) with detection hits lands in `needs_user_action` state rather than attempting to render interactive nudges.
- **rac-v1.AC4.6 Edge:** Two independent firings of the same rule on one submission produce two distinct `detection_finding` rows.

### rac-v1.AC5: Build + scan pipeline enforces the severity gate

- **rac-v1.AC5.1 Success:** A clean golden repo builds, is pushed to ACR, passes both Grype and Defender, and the submission advances to `awaiting_research_review`.
- **rac-v1.AC5.2 Failure:** A golden repo with a planted HIGH-severity CVE in a dependency is blocked; submission transitions to `scan_rejected`; researcher sees the specific CVE list in the UI.
- **rac-v1.AC5.3 Failure:** A repo with an invalid Dockerfile transitions to `pipeline_error` with the build log artifact accessible from the submission detail view.
- **rac-v1.AC5.4 Edge:** A Defender scan that exceeds its timeout produces a partial verdict; submission advances with an explicit "Defender scan pending" warning badge surfaced to the IT approver.
- **rac-v1.AC5.5 Edge:** The layer cache is hit on a second build of an unchanged repo (evidenced by cache-hit messages in the build log or substantially reduced wall-clock time).

### rac-v1.AC6: Approved apps deploy automatically

- **rac-v1.AC6.1 Success:** Final IT approval triggers Tier 3 provisioning: an ACA app is created with the correct image digest, env vars, scaling rules (`min-replicas=0`), and tags; a DNS A-record is inserted; per-app signing key is created in Key Vault; submission transitions to `deployed`.
- **rac-v1.AC6.2 Success:** The app resolves at `https://<slug>.${PARENT_DOMAIN}`; when no replicas are warm, the shim serves the "waking up" interstitial, wakes the app, and redirects to the original URL within the configured wake budget.
- **rac-v1.AC6.3 Failure:** A provisioning error (e.g., DNS conflict, quota exhaustion) surfaces in the admin UI with a retry control; the submission remains `approved` but not `deployed` until resolution.
- **rac-v1.AC6.4 Edge:** A re-submitted app (new approved submission for an existing slug) updates `app.current_submission_id` atomically; the previous image is retained in ACR with the old `submission_id` tag so history is recoverable.

### rac-v1.AC7: Reviewer access works via tokenized URLs

- **rac-v1.AC7.1 Success:** A valid token presented via URL query sets an HttpOnly/Secure/SameSite=Lax cookie; shim redirects to the clean URL (token stripped); the access is written to `access_log`.
- **rac-v1.AC7.2 Success:** A revoked token (`jti` present in `revoked_token`) is rejected within 60 seconds of revocation and the user sees the branded "revoked" page.
- **rac-v1.AC7.3 Success:** An expired token produces the branded "expired" page with the researcher's contact email and PI name.
- **rac-v1.AC7.4 Failure:** A malformed, forged, or wrong-audience token returns a generic 403 page that does not disclose the specific validation failure.
- **rac-v1.AC7.5 Edge:** An app in `access_mode=public` serves all requests without token validation; `access_log` rows are still written with `token_jti=NULL`.
- **rac-v1.AC7.6 Edge:** A valid token issued for App A is rejected when presented at App B (audience-claim mismatch).

### rac-v1.AC8: Asset handling supports v1 kinds

- **rac-v1.AC8.1 Success:** An `upload` asset provided via the submission form is stored in Blob, its sha256 is computed server-side and persisted in `asset`, and it is mounted at the declared `mount_path` inside the deployed ACA app.
- **rac-v1.AC8.2 Success:** An `external_url` asset with a reachable URL and matching sha256 is fetched, verified, cached in Blob, and mounted.
- **rac-v1.AC8.3 Failure:** An `external_url` asset whose fetched content does not match its declared sha256 blocks deployment; the IT approver sees the mismatch in the approval UI with both expected and actual hashes.
- **rac-v1.AC8.4 Failure:** An `external_url` asset whose URL is unreachable puts the submission in `needs_user_action` with a clear explanation.
- **rac-v1.AC8.5 Edge:** A submission without a committed `rac.yaml` but with form-declared assets produces an equivalent parsed manifest internally, indistinguishable downstream from a researcher-committed manifest.
- **rac-v1.AC8.6 Edge:** A manifest that declares a `shared_reference` asset is rejected at submission time in v1 with a "shared references coming soon" message naming the specific entry.

### rac-v1.AC9: Ownership model is wired

- **rac-v1.AC9.1 Success:** A submission captures `pi_principal_id` (validated as a real Entra principal via Graph) and `dept_fallback` as required fields.
- **rac-v1.AC9.2 Success:** The nightly Graph sweep detects a PI whose Entra account is deactivated and flags every app owned by that PI with a "Owner deactivated — dept review pending" status visible to admins.
- **rac-v1.AC9.3 Edge:** An app's `pi_principal_id` can be transferred without losing audit history; historical `approval_event` rows retain their original `actor_principal_id` unchanged.

### rac-v1.AC10: Observability is operational

- **rac-v1.AC10.1 Success:** The shim writes an `access_log` row for every proxied request, including requests served under `access_mode=public`.
- **rac-v1.AC10.2 Success:** The custom metrics listed in the Architecture section (`rac.submissions.by_status`, `rac.shim.token_validations`, `rac.shim.wake_up_duration_ms`, `rac.scans.verdict`, `rac.approvals.time_to_decision_seconds`) emit to App Insights on their respective events with the documented label dimensions.
- **rac-v1.AC10.3 Success:** The pager-tier alert for shim 5xx rate > 1% over 5 minutes fires in response to a controlled fault injection and pages the on-call channel.
- **rac-v1.AC10.4 Success:** Control Plane and Shim logs are structured JSON with stable fields (`submission_id`, `app_id`, `principal_id`, `request_id`) in Log Analytics.
- **rac-v1.AC10.5 Edge:** The Event Hub export surface exists per `docs/runbooks/siem-export.md` and can be subscribed to by a test consumer without any code changes.

### rac-v1.AC11: Cost attribution works

- **rac-v1.AC11.1 Success:** Every Tier 2 and Tier 3 Azure resource carries `rac_app_slug`, `rac_pi_principal_id`, `rac_submission_id` (where applicable), and `rac_env` tags at creation time.
- **rac-v1.AC11.2 Success:** The daily Azure Cost Management export populates `cost_snapshot_monthly`; the admin dashboard renders per-app monthly cost with an explicit month-to-date estimate.
- **rac-v1.AC11.3 Edge:** An app that scales to zero for an entire billing period still produces a `cost_snapshot_monthly` row reflecting the minimal shared ACA environment overhead attributed to it.

### rac-v1.AC12: Cross-cutting audit and error hygiene

- **rac-v1.AC12.1:** All append-only tables (`access_log`, `approval_event`, `revoked_token`, and `detection_finding` insert path) have no `UPDATE` or `DELETE` grants to application database roles in production.
- **rac-v1.AC12.2:** Every API error response body includes a `correlation_id` that is also present in the corresponding App Insights trace for that request.
- **rac-v1.AC12.3:** API error responses never leak Postgres error text, stack traces, or internal URIs to the client.

## Glossary

- **RAC**: Research Application Commons. The platform described in this document — a governed hosting surface for researcher-authored containerized applications at a single institution.
- **MRAC**: Earlier name for the project (Moffitt Research Application Commons), used before the 2026-04-23 rename to the institution-agnostic "RAC."
- **BBSR**: Bioinformatics and Biostatistics Shared Resource — the Moffitt core that operates and maintains the RAC deployment.
- **ACA**: Azure Container Apps. Microsoft's serverless container hosting service, used here for the Control Plane, Shim, and researcher apps. Supports scale-to-zero and internal-only ingress.
- **ACR**: Azure Container Registry. Stores built researcher container images; also the trigger point for Defender for Containers auto-scans.
- **Azure Front Door Premium**: Microsoft's global edge service providing WAF (Web Application Firewall), DDoS absorption, and TLS termination at the CDN layer, upstream of the regional App Gateway.
- **App Gateway v2**: Azure Application Gateway — a regional Layer-7 load balancer with WAF capability. In this topology it performs hostname-based routing for `*.${PARENT_DOMAIN}` after Front Door.
- **Defender for Containers**: Microsoft Defender product that automatically scans container images pushed to ACR for known CVEs. RAC's pipeline polls this scan alongside the Grype scan and consolidates both verdicts against the configured severity gate.
- **Key Vault**: Azure Key Vault. Stores per-app JWT signing keys, HMAC pipeline callback secrets, and other deployment secrets. The Shim caches public keys from Key Vault for token validation.
- **Bicep**: Microsoft's declarative IaC language for Azure (a typed abstraction over ARM templates). RAC uses Bicep for Tier 2 static infrastructure, deployed via GitHub Actions.
- **Entra ID**: Microsoft's cloud identity platform (formerly Azure Active Directory). Provides OIDC-based single sign-on for researchers and approvers, and the Graph API for principal lookups.
- **Graph API**: Microsoft Graph — the REST API over Entra ID used here to validate PI principal IDs at submission time and to detect deactivated accounts in the nightly ownership sweep.
- **OIDC**: OpenID Connect. The identity layer on top of OAuth2 used for interactive researcher and approver authentication against the institution's Entra tenant.
- **JWT**: JSON Web Token. A signed, self-describing token. RAC uses JWTs for reviewer access tokens (signed with per-app keys stored in Key Vault); the Shim validates signature, expiry, audience, and denylist membership on every request.
- **HMAC**: Hash-based Message Authentication Code. Used to authenticate callbacks from the build pipeline to the Control Plane, and for outgoing webhook deliveries to registered agents, so the receiver can verify the payload has not been tampered with.
- **SBOM**: Software Bill of Materials. A machine-readable inventory of all packages and layers in a container image, generated here by Syft and stored in Blob alongside each scan result.
- **Manifest (`rac.yaml`)**: A YAML file at the repo root declaring a submission's non-code dependencies (assets). Can be committed by power users for reproducibility or auto-generated from the submission form for novice researchers.
- **Scale-to-zero**: An ACA configuration (`min-replicas=0`) where a container app runs no instances when idle and is started on first request. Researcher apps all use this to reduce cost; the Shim's cold-start interstitial handles the resulting startup latency.
- **Token-Check Shim**: A lightweight reverse proxy (`min-replicas=1`) that is the single public entry point to all researcher ACA apps. It validates JWT reviewer tokens, sets HttpOnly cookies, serves branded wake-up and error pages, and writes per-request audit rows.
- **Denylist**: A table (`revoked_token`) of revoked JWT identifiers (`jti`). The Shim caches this list with a 60-second TTL, providing near-real-time revocation without a synchronous database call on every request.
- **Tier 1 / Tier 2 / Tier 3 split**: A deliberate three-level separation of infrastructure by change frequency and operator. Tier 1 is one-time manual tenant bootstrap; Tier 2 is static platform IaC managed in Bicep via CI; Tier 3 is dynamic per-submission provisioning done at runtime by the Control Plane via the Azure SDK.
- **Grype**: An open-source container vulnerability scanner (from Anchore). Runs in the build pipeline against the pushed image and produces a CVE report with EPSS and KEV annotations.
- **Syft**: An open-source SBOM generator (from Anchore). Runs in the pipeline alongside Grype to produce a package inventory that is archived in Blob with each submission's scan artifacts.
- **EPSS / KEV**: Exploit Prediction Scoring System and Known Exploited Vulnerabilities catalog — contextual signals attached to CVEs in the Grype report to help prioritize findings beyond raw CVSS severity.
- **Severity gate (`SCAN_SEVERITY_GATE`)**: A deployment-time configuration value specifying the minimum CVE severity level that will block a submission. Evaluated against consolidated Grype + Defender findings before the submission can advance to research review.
- **Functional core, imperative shell (FCIS)**: An architectural pattern separating pure business logic (state-machine transitions, validation, token checks) from side-effecting adapters (database, Azure SDK, HTTP). Used in both the Control Plane and Shim to make core logic testable without infrastructure.
- **Idempotency key**: A client-supplied identifier on a write request (`POST /submissions`) that allows the server to return the original result on retries rather than creating duplicate records.
- **`repository_dispatch`**: A GitHub Actions mechanism for externally triggering a workflow via the GitHub API. The Control Plane uses this to dispatch the build-and-scan pipeline workflow in the separate `rac-pipeline` repo.
- **UUIDv7**: A UUID variant whose first bits encode a timestamp, giving naturally time-ordered primary keys without a separate sequence. Used for all RAC database primary keys.
- **Pydantic**: A Python data validation library. Used in the Control Plane to define typed request/response schemas that also drive OpenAPI documentation generation.
- **Alembic**: A Python database migration tool for SQLAlchemy. Manages schema version history for the RAC Postgres database.

## Architecture

RAC is a single-tenant-per-deployment platform. Each deploying institution runs its own copy in its own Azure subscription; the codebase is institution-agnostic with deployment-time configuration (parent domain, IdP, branding, approver roles, scanner thresholds). No shared database, no `tenant_id` columns, no cross-tenant logic. Moffitt Cancer Center is the first deployment.

### Components

**Control Plane** (`apps/control-plane/`, FastAPI + React). The administrative web app and submission API. Owns submissions, approvals, ownership records, reviewer-token minting, cost dashboards, and Tier 3 provisioning. Authenticates humans via OIDC; accepts service-to-service clients via OAuth2 client credentials. Issues HMAC-signed webhook callbacks to registered agents.

**Token-Check Shim** (`apps/shim/`, Go or Python). A small reverse proxy with `min-replicas=1`. The only public path to researcher apps — researcher ACA apps have internal ingress only. Validates JWT reviewer tokens (signature, expiry, audience, denylist), sets HttpOnly cookies after first use, serves a branded "waking up" interstitial on cold start, writes an `access_log` row per request, renders error pages for expired/revoked/invalid tokens.

**Build + Scan Pipeline** (`rac-pipeline` GitHub repo; `pipeline/workflows/build-and-scan.yml`). Dispatched by the Control Plane via `repository_dispatch` on each submission. Clones the researcher repo, builds with a buildx layer cache, pushes to ACR (triggering Defender auto-scan), generates an SBOM via Syft, runs Grype, consolidates findings, and posts an HMAC-signed callback to the Control Plane.

**Pre-submission detection rules** (`apps/control-plane/src/detection/rules/`). Versioned Python modules implementing a stable `Rule` contract. Run before the pipeline is dispatched; findings become interactive nudges in the researcher's UI or put service-to-service submissions into `needs_user_action`. Every firing logs `rule_id@version` + decision + eventual outcome for empirical rule tuning.

**Researcher apps** (runtime-provisioned ACA apps, one per approved submission). Internal ingress only, reachable solely through the Shim. `min-replicas=0`. Provisioned, updated, and retired by the Control Plane at approval time using the Azure SDK.

### Three-tier infrastructure split

This is deliberately not a single tool doing everything. The tiers have different change frequencies and operational concerns.

| Tier | What it does | Tool | When it runs |
|---|---|---|---|
| **Tier 1** | One-time tenant bootstrap (subscription, Entra app registrations, service principals, DNS delegation, initial RBAC) | Azure Portal + scripted `az` commands, documented in `docs/runbooks/bootstrap.md` | Once per deployment, by a cloud admin |
| **Tier 2** | Static platform infrastructure (VNet, ACA environment, ACR, Postgres, Key Vault, Front Door, App Gateway, Log Analytics, App Insights, Control Plane + Shim ACA app shells) | Bicep modules in `infra/` deployed via GHA workflow `infra/deploy.yml`, human-gated promotion dev → staging → prod | On IaC PR merge |
| **Tier 3** | Dynamic per-submission resources (researcher ACA apps, DNS records, per-app Key Vault signing keys, asset storage mounts) | Azure SDK for Python (`azure-mgmt-appcontainers`, `azure-mgmt-dns`, `azure-keyvault-*`) called from the Control Plane | On final approval, at request time |

Tier 3 uses the SDK directly rather than generating Bicep at runtime because IaC round-trips belong in a CI pipeline, not a web request handler. Pulumi Automation API is the documented swap target if runtime IaC guarantees become necessary later.

### Data plane

Single Postgres schema, single-tenant. Core tables:

- `submission` — one row per submitted version; immutable after creation except for status/terminal timestamps
- `app` — one row per researcher application; 1:1 with the currently-deployed submission
- `asset` — per-submission declared non-code dependencies (`upload`, `external_url`, `shared_reference` kinds)
- `scan_result` — consolidated pipeline verdicts with findings, SBOM references, scanner artifact URIs
- `detection_finding` — every pre-submission rule firing with rule_id@version + researcher decision + outcome
- `approval_event` — append-only audit of every approval-chain action
- `reviewer_token` — JWT claims and metadata; revocation via separate `revoked_token` table
- `access_log` — append-only, written by the shim
- `signing_key_version` — per-app JWT signing key rotation tracking
- `agent` + `webhook_subscription` — submission front-ends (web UI, ServiceNow, CLI) and their callback registrations
- `cost_snapshot_monthly` — materialized from daily Azure Cost Management exports
- `shared_reference_catalog` — institutional curated reference assets (schema present; UI v2)

Conventions: snake_case, UUIDv7 primary keys, `created_at`/`updated_at` timestamptz, JSONB only for genuinely variable-shape payloads, FKs `ON DELETE RESTRICT` (archive, don't cascade-delete). Append-only tables have no `UPDATE` grants.

Large artifacts (SBOMs, scan reports, researcher uploads, Cost Management exports) live in Blob Storage and are referenced by URI.

### Submission status finite state machine

```
awaiting_scan ──(pipeline error)──▶ pipeline_error   (retryable)
      │
      ├─(severity gate failed)──▶ scan_rejected
      │
      ▼
needs_user_action ──(researcher asks help)──▶ needs_assistance ──▶ awaiting_research_review
      │
      ▼
awaiting_research_review ──(reject)──▶ research_rejected
      │
      ▼
awaiting_it_review ──(reject)──▶ it_rejected
      │
      ▼
approved ──(provisioning completes)──▶ deployed   (terminal)
```

Invariants: at most one non-terminal submission per `app.slug`; every submission is preserved; `app.current_submission_id` always references a submission in terminal `deployed` state.

### Reviewer access flow

1. Researcher opens the app in the Control Plane, mints a reviewer token (label, expiry up to 180 days). Receives a one-shot URL: `https://<slug>.${PARENT_DOMAIN}/?rac_token=<jwt>`.
2. Reviewer clicks. The shim validates signature, expiry, issuer, audience, and denylist membership. Sets HttpOnly/Secure/SameSite=Lax cookie, 302-redirects to clean URL.
3. Subsequent requests ride the cookie. Every request writes an `access_log` row.
4. If the target app has no warm replicas, the shim returns a branded "starting up" interstitial with a JS-triggered wake call; reviewer sees deliberate 5–10s wait, not a browser spinner.
5. Post-publication, researcher or admin may set `app.access_mode = public`. The shim skips token validation and adds a visible "🌐 Public" banner to the interstitial.

Revocation: insert into `revoked_token`. Shim caches the denylist for 60 seconds; revocation is effective within that window.

### Build + scan pipeline flow

```
Control Plane dispatches GHA workflow with payload:
  { submission_id, repo_url, ref, dockerfile_path,
    callback_url, callback_hmac_secret_ref }
      │
      ▼
GHA workflow:
  1. Clone researcher repo @ ref
  2. docker buildx build, tag <slug>:<submission_id>
     (ACR-backed layer cache keyed on repo_url + dockerfile_path)
  3. docker push → Defender auto-scan begins
  4. syft → SBOM
  5. grype → vuln report with EPSS + KEV
  6. Poll Defender result (bounded wait)
  7. Evaluate vs SCAN_SEVERITY_GATE
  8. Archive SBOM, scanner JSON reports, build log to Blob
  9. HMAC-signed POST to callback_url with verdict + artifact URIs
      │
      ▼
Control Plane verifies HMAC, updates submission.status
```

Failure modes: `build_failed`, `scan_rejected`, `pipeline_error` (retryable). Re-submission creates a new `submission` row; submissions are immutable.

### Submission manifest

`rac.yaml` at repo root declares non-code dependencies. Three asset kinds:

- **upload**: researcher uploaded a file at submission time; stored in Blob, hash computed server-side
- **external_url**: platform fetches at build-finalize with mandatory sha256 check, caches in Blob, mounts read-only
- **shared_reference**: reference to platform-curated catalog entry (schema only in v1; UI in v2)

The manifest is generated by the submission form for novice researchers (who never see YAML) and can be committed directly by power users (reproducibility). Both paths produce the same parsed structure stored in `submission.manifest`.

### Observability and ops

- Single Log Analytics workspace per environment; App Insights for Control Plane and Shim; all logs structured JSON with stable fields (`submission_id`, `app_id`, `principal_id`, `request_id`).
- Custom metrics: `rac.submissions.by_status`, `rac.shim.token_validations{result}`, `rac.shim.wake_ups`, `rac.shim.wake_up_duration_ms`, `rac.scans.verdict{verdict}`, `rac.approvals.time_to_decision_seconds`.
- Pager-tier alerts: shim 5xx > 1% over 5 min, Control Plane 5xx > 1% over 5 min, Postgres connection failures, Key Vault access denied, pipeline workflow stuck > 2× timeout.
- SIEM readiness: `access_log` is append-only; all logs JSON-structured; Event Hub is documented as the export point (connector itself is v2).
- Single region, zone-redundant. Multi-region is permanently out of scope.

### Cost attribution

Every Tier 2 and Tier 3 Azure resource is tagged with `rac_app_slug`, `rac_pi_principal_id`, `rac_submission_id`, `rac_env`. Azure Cost Management exports daily to Blob; a Control Plane job aggregates into `cost_snapshot_monthly`. Admin dashboard reads from this table.

## Existing Patterns

This is a greenfield project. No existing codebase, no prior patterns to inherit. The design introduces these new patterns, chosen deliberately for this project and documented here for consistency as the codebase grows:

- **Functional core, imperative shell** (FCIS) separation in both the Control Plane and the Shim. Pure functions for state-machine transitions, token validation, slug generation, manifest parsing, rule evaluation; side-effecting adapters for Postgres, Azure SDK, and HTTP calls.
- **Service structure** follows the house convention: domain logic in `src/services/<domain>/`, API routing in `src/api/routes/`, database access in `src/data/`.
- **Append-only audit tables** with no `UPDATE`/`DELETE` grants in production.
- **Typed API boundaries** via Pydantic (Control Plane) with OpenAPI auto-generation; request/response contracts are colocated with routes.
- **IaC in Bicep** for Tier 2 static infrastructure; Azure SDK for Tier 3 dynamic provisioning; manual runbook for Tier 1 tenant bootstrap.
- **Versioned rules as code** for pre-submission detection: rules ship as Python modules in `apps/control-plane/src/detection/rules/`, not as database rows — low volume, high change scrutiny.

Every new module or service should align with these patterns unless explicitly justified otherwise in the PR.

## Additional Considerations

**Scale expectations.** Year-one targets are 10–30 apps total with fewer than 5 warm concurrently. Nothing in the design is sized for web-scale traffic. Front Door + App Gateway are chosen for the paved-road operational story, not for performance headroom; if scale stays at pilot levels for years, the ongoing cost of those two services remains a minor but real operational line item that's worth periodic review.

**Security boundaries worth being explicit about.**

- Researcher apps have internal ingress only. The Shim is the single public entry point. This assumption underpins the access-control model; any future deviation (e.g., exposing a researcher app directly for a specific integration) must go through a design review.
- `docker build` during the pipeline runs in ephemeral GHA runners with scoped secrets. Egress is not hardened in v1; future work may add egress restrictions to the build step. A malicious Dockerfile can make outbound calls during build — we accept this risk in exchange for runner simplicity.
- Per-submission HMAC secrets are scoped to 2× the pipeline timeout and stored in Key Vault; no long-lived tokens floating in GHA variables.
- Per-app JWT signing keys are scoped to that app. Rotation is annual with a 30-day overlap window; shim accepts tokens signed by either key during overlap.

**Rule engine evolution.** Detection rules start as `warn` severity by default. Graduation to `error` (blocking) happens only after data shows a rule is consistently correct. The analytics substrate (firing counts per rule, acceptance rates, correlation with deploy outcomes) is first-class in the schema precisely so rules can be evaluated empirically and revised rather than rubber-stamped forever.

**Thornier operational concerns deferred by name.**

- **Asset upload UX for multi-gigabyte files.** Resumable upload with recovery on network blips is a hard UX problem. Tus.io or Azure SDK resumable multipart on the server; React-Dropzone-style UI on the front. The design provides the server-side pieces; genuinely polished client UX is a post-v1 refinement.
- **Defender scan latency is variable.** Microsoft doesn't provide an SLO on scan completion. The pipeline's timeout-with-partial-verdict path handles this, but the IT approver UI must clearly indicate when a decision is being made on a Grype-only verdict.
- **Long-running builds (>60 min).** Some legitimately large scientific containers hit the ceiling. The signal is usually that the researcher is baking data into the image that should be an asset; Phase 4 detection rules flag the obvious cases. True exceptions become a conversation with the submitter, not a design change.

**ServiceNow readiness.** No connector ships in v1, but the architecture is structurally ready: submission is API-first, agents and webhook subscriptions are modeled, actor-on-behalf-of semantics are in the data model, idempotency keys are honored. A ServiceNow adapter becomes a plugin when a real integration project exists; the design does not preclude it.

**SIEM readiness.** No connector ships in v1. Design decisions made to keep this cheap later: structured JSON logging everywhere with stable field names; `access_log` and `approval_event` are pure append-only; Event Hub is identified as the export point in the runbooks. Logs contain no PII beyond Entra principal IDs to keep the eventual SIEM integration simple.

**Implementation scoping.** This design has exactly 8 phases, which is the hard limit for a single implementation plan. If further work (`shared_reference` catalog UI, ServiceNow connector, CLI client, dry-run endpoint, SIEM connector) is undertaken, a second implementation plan will be required.
